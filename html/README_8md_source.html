2 # bidirectional_map
3 Implementation of a bidirectional associative container in c++. Its goal is to behave
4 similarly to popular stl containers like `std::unordered_map` while providing efficient
5 lookup from key to value as well as from value to key.
6 ## Properties
7 The `bidirectional_map` container contains pairs of values of type K1 and K2.
8 * Objects in the container are immutable, neither values of type K1 nor values of type
9  K2 can be modified to ensure the integrity of the underlying associative containers
10 * The container supports the use of different associative containers as base. The
11  default base container is `std::unordered_map` for both forward and inverse lookup.
12  Other tested containers are `std::map` as well as `std::multimap` and
13  `std::unordered_multimap`.
14 * The mapping from values of K1 to values of K2 is enforced to be injective if the
15  underlying containers for both forward and inverse lookup contain unique keys (like
16  in the default case). This means that for example two pairs (k1, k2) and (k1', k2')
17  can only be inserted at the same time if k1 != k1' **and** k2 != k2'. The use of
18  multimaps as base containers relaxes this constraint.

24 ## Code Example
25 An instance of `bidirectional_map` can be created similarly to `std::unordered_map`:
26 ~~~{.cpp}
27 #include <string>
28 #include <unordered_map>
29 #include "bidirectional_map.hpp"
30 
31 // empty container
32 bimap::bidirectional_map<std::string, int> map;
33 
34 // using initializer list
35 bimap::bidirectional_map<std::string, int> map1 = {{"Test", 1}, {"Hello", 2}};
36 
37 // from same container type
38 bimap::bidirectional_map<std::string, int> map3(map1.begin(), map1.end());
39 
40 // from different container type
41 std::unordered_map<std::string, int> values = {{"abc", 1}, {"def", 2}};
42 bimap::bidirectional_map<std::string, int> map2(values.begin(), values.end());
43 ~~~
44 From the items used for initialization only unique ones are inserted (see properties).
45 Further items can be inserted using the `emplace` method
46 ~~~{.cpp}
47 bimap::bidirectional_map<std::string, int> map;
48 map.emplace("NewItem", 17); // constructs new item in place
49 
50 // no insertion. 17 already exists in the container. Returns iterator to ("NewItem", 17)
51 auto [iterator, inserted] = map.emplace("AnotherItem", 17);
52 ~~~
53 Item lookup:
54 ~~~{.cpp}
55 bimap::bidirectional_map<std::string, int> map{{"NewItem", 12}, {"Stuff", 17}};
56 auto location = map.find("NewItem");
57 if (location != map.end()) {
58  std::cout << location->first << std::endl;
59 }
60 
61 if (map.contains("Stuff")) {
62  std::cout << map.at("Stuff") << std::endl; // at is only available when using underlying container that enforces
63  // unique keys
64 }
65 ~~~

67 ### Inverse Access
68 Using the `inverse()` member, inverse lookup and insertion is possible:
69 ~~~{.cpp}
70 bimap::bidirectional_map<std::string, int> map;
71 map.inverse().emplace(123, "one two three"); // inverse insertion
72 auto invLocation = map.inverse().find(123); // inverse lookup
73 std::cout << invLocation->first << std::endl; // prints '123'
74 
75 // inverse of inverse() is again the original
76 auto location = map.inverse().inverse().find("one two three");
77 ~~~
78 `inverse()` returns a reference to `bidirectional_map` where the template types K1 and K2
79 are reversed. It behaves exactly like the original map except... well the other way around.
80 Even the iterator members are reversed. Copying the `inverse()` container is allowed and
81 will copy the container contents. Moving from `inverse()` is also allowed and behaves as
82 expected.
83 ~~~{.cpp}
84 bimap::bidirectional_map<std::string, int> map; // map from std::string -> int
85 auto inverse = map.inverse(); // independent (copied) container of reversed type (int -> string)
86 auto &inverseRef = map.inverse(); // inverse access to the same container
87 ~~~

89 ### Custom Map Base Container
90 It is possible to specify a custom map base container for forward lookup as well as for
91 inverse lookup. The default map base type is `std::unordered_map` for forward access as
92 well as for inverse access. Another possible map base type is `std::map`:
93 ~~~{.cpp}
94 // only forward access uses the ordered map std::map.
95 // Inverse access is till provided through std::unordered_map
96 bimap::bidirectional_map<std::string, int, std::map> map;
97 
98 // Both forward and inverse access use std::map
99 bimap::bidirectional_map<std::string, int, std::map, std::map> map1;
100 ~~~
101 Another scenario for using a different map base type is when you need to specify for
102 example a custom hash function:
103 ~~~{.cpp}
104 struct MyString {...}; // Custom data structure with no default std::hash specialization
105 struct MyHash {...}; // Custom hash struct
106 struct MyComparator {...}; // Custom comparator necessary for std::unordered_map
107 
108 template<typename T, typename U>
109 using BaseMap = std::unordered_map<T, U, MyHash, MyComparator>;
110 
111 // for inverse access the default std::unordered_map is sufficient
112 bimap::bidirectional_map<MyString, int, BaseMap> map;
113 ~~~
